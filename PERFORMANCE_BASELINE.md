## Bottleneck Analysis

### 1. Authentication Overhead
- `list_models` with API key auth: ~187 RPS vs `list_models_no_auth`: ~533 RPS
- Authentication adds ~3x overhead due to API key validation logic
- `login_flow` is the slowest at 151 RPS (bcrypt password verification is CPU-intensive by design)

### 2. Concurrency Scaling
- At 10 concurrent: ~474 RPS (health), latency P50 ~12ms
- At 50 concurrent: ~283 RPS (health), latency P50 ~102ms
- At 100 concurrent: ~119 RPS (health), latency P50 ~422ms
- **Single-worker uvicorn becomes the bottleneck above 50 concurrent connections**

### 3. Key Observations
- All scenarios maintain **100% success rate** even under stress (no errors/crashes)
- SQLite performs well for read-heavy workloads (health, models list)
- Login (bcrypt) is intentionally slow for security -- not a bug

## Optimization Recommendations

1. **Multi-Worker Deployment**: Production should use `uvicorn --workers N` (N = CPU cores)
   - Expected improvement: ~linear scaling up to CPU core count
2. **Connection Pool (PostgreSQL)**: Switch to PostgreSQL with connection pooling for concurrent writes
3. **Response Caching**: `/v1/models` result can be cached with TTL (model list changes infrequently)
4. **API Key Lookup**: Consider in-memory hash map for API key validation (avoid DB hit per request)
5. **Reverse Proxy**: Deploy behind Nginx/Caddy for TLS termination + connection buffering
6. **Async bcrypt**: Login flow uses async bcrypt already -- good. Consider rate-limiting as mitigation.
