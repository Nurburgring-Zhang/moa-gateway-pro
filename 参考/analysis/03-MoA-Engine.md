## 一、项目概述

### 1.1 项目身份

**MoA Engine (Mixture of Agents Engine)** 并非传统的可执行代码框架,而是一套**注入大语言模型内部的"元提示词操作系统"**(Meta-Prompt Operating System)。它通过一组严格的思维协议,强制将单一 LLM 转变为"分工明确、相互对抗、最终熔铸重构"的虚拟专家团队,在单模型上涌现出**超越单线思维极限**的高阶智能。

> **核心隐喻**: "单个模型的'思考'可以涌现智能;而一群模型的'组织'可以涌现更高阶的智能。" (README L3)

### 1.2 解决的问题

| 单模型痛点 | MoA 解法 |
|-----------|---------|
| 自回归单线思维,容易"一路狂奔" | 角色分化强制上下文切换,模拟多进程 |
| 确认偏误,自我反思失效 | 结构化对抗制造"认知摩擦" |
| 多角色提示词沦为"观点拼盘" | 熔铸决策者强制高阶再创造 |
| 全栈盲区,样样通样样松 | 垂直分工,每个专家绑定细分领域 |

### 1.3 部署形态

- **零代码、纯 Prompt 驱动**: 复制元提示词 → 粘贴到任意长上下文 LLM → 即插即用
- **跨模型兼容**: GPT-4, Claude 3.5 Sonnet, Gemini 1.5 Pro 等
- **打包形式**: `.skill` 文件 (ZIP 格式) 用于技能分发

## 三、详细能力列表

### A. API 与协议层 (API / Protocol)

#### A1. 元提示词编排协议
- **位置**: `README.md` L56-102, `references/moa-meta-prompt.md` L37-85
- **能力**: 提供可直接复制粘贴的**完整元 Prompt 模板**,包含核心信条、动态角色原型、任务执行协议、输出格式约束
- **关键字段**: `{用户任务}` 占位符用于任务注入
- **接口契约**: 输入 = 任务描述, 输出 = 4 阶段结构化推演 + `[最终答案]`

#### A2. 4 阶段强制执行协议
- **位置**: `SKILL.md` L41-128, `README.md` L25-46
- **能力**: 协议强制规定不可跳过、不可合并的 4 个显式阶段
  - 阶段 1: 动态编排与解构 (战略规划师)
  - 阶段 2: 并行深度推演 (领域专家群)
  - 阶段 3: 结构化对抗与反思 (批判者 ↔ 专家 1-2 轮)
  - 阶段 4: 逻辑熔铸与终局裁决 (熔铸决策者)
- **约束**: 全程显式输出, 不可隐藏思考过程, 必须以 `[最终答案]:` 开头收尾

#### A3. 角色选择 API (动态指派)
- **位置**: `SKILL.md` L33-37
- **能力**: 战略规划师根据任务特征**动态选派 4-6 个具体角色**,而非固定角色列表
- **规则**: 角色必须具体到细分领域 (如"Three.js 3D渲染专家"而非"前端开发")
- **配置**: 每个子任务 = 1-2 名专家 + 1 名批判者

#### A4. 阶段输出格式契约
- **位置**: `references/moa-meta-prompt.md` L137-179
- **能力**: 规定 4 个阶段各自的输出文本格式,确保可追溯性
  - 阶段 1: `[战略规划师]: 任务分解如下:`
  - 阶段 2: `[专家角色名]: 我的方案如下...`
  - 阶段 3: `[批判者]: 针对[专家A]的方案...` + `[专家A]: 回应批判...`
  - 阶段 4: `[熔铸决策者]:` + `[最终答案]:`

#### A5. 思维模式切换 API
- **位置**: `SKILL.md` L45
- **能力**: 在阶段 2-3 强制**发散思维**;在阶段 4 强制**严谨工程师模式**进行裁决
- **机制**: 通过角色身份切换强制模型内部系统设定变更

### B. 数据模型 (Data Model)

#### B1. 任务分解树模型
- **位置**: `SKILL.md` L51-63, `references/moa-system-guide.md` L83-95
- **结构**:
```
任务 (Task)
├── 子任务 A (高内聚、低耦合)
│   ├── 专家: [具体头衔] -- 核心产出: [目标]
│   └── 批判者: [具体头衔]
├── 子任务 B
│   ├── 专家: [具体头衔] -- 核心产出: [目标]
│   └── 批判者: [具体头衔]
└── ...
```
- **原则**: 分解粒度可独立交付, 叶子节点必须有"专家 + 批判者"对偶

#### B2. 四类核心角色实体
- **位置**: `SKILL.md` L26-31
| 角色类型 | 思维模式 | 关键约束 |
|---------|---------|---------|
| 战略规划师 | 任务规划型 | 全局视野, 分解粒度可独立交付 |
| 领域专家 | 生成者 | 绑定具体子任务, 不可跨域输出 |
| 无情批判者 | 批判者 | 不接受空泛否定, 必须可操作 |
| 熔铸决策者 | 综合者 | 禁止罗列, 必须再创造 |

#### B3. 方案版本控制模型
- **位置**: `SKILL.md` L92-96, `references/moa-case-study.md` L88-104
- **能力**: 专家方案支持迭代版本化 (修正方案 v1 → v2 → ...)
- **机制**: 每个批判 → 修正对应, 形成"攻击-回应"对偶

#### B4. 认知摩擦回路模型
- **位置**: `references/moa-system-guide.md` L154-176
- **结构**: 批判者 → 具体攻击 → 专家 → 逐条回应 → 修正方案 v2 → 再次审视
- **终止条件**: 批判者明确"已无重大结构性风险"

### C. 算法 (Algorithm)

#### C1. 任务分解算法
- **位置**: `references/moa-system-guide.md` L83-95
- **输入**: 模糊复杂任务
- **输出**: 高内聚、低耦合子任务树
- **评价标准**:
  - 低质量分解: 写游戏 → "写作模型" + "代码模型"
  - 高质量分解: 写游戏 → 玩法设计 + 剧情 + 前端 + 后端 + 测试

#### C2. 角色匹配算法
- **位置**: `SKILL.md` L34-37
- **输入**: 子任务 + 能力维度
- **输出**: 具体专家头衔 + 批判者头衔
- **规则**:
  1. 战略规划师先识别所需能力维度
  2. 为每个子任务匹配最具体的专家头衔
  3. 批判者可以跨子任务审视,但必须指名道姓

#### C3. 认知摩擦对抗算法
- **位置**: `references/moa-system-guide.md` L131-146
- **算法**:
  1. 批判者阅读所有专家产出
  2. 发起系统性攻击 (必须包含: 哪里不好 + 为什么 + 在什么条件下)
  3. 专家逐条对应回应
  4. 迭代 1-2 轮
  5. 终止: 批判者确认"已无重大结构性风险"

#### C4. 信息高保真传递算法
- **位置**: `references/moa-system-guide.md` L130-146
- **核心要求**:
  - 低质量: "这段代码有问题"
  - 高质量: "在并发场景下,第45行的共享变量存在竞态条件风险"
- **保障机制**: 共享上下文, 不可私下传递, 累积式改进

#### C5. 冲突消解算法
- **位置**: `references/moa-system-guide.md` L162-176
- **流程**:
  1. 不靠投票 (避免丢失少数派洞见)
  2. 由熔铸决策者主持虚拟辩论
  3. 总结争论焦点
  4. 要求双方提供论据
  5. 基于逻辑和证据裁决
  6. 记录决策理由
- **本质**: 冲突是催化剂,不是缺陷

#### C6. 三次认知跃迁算法
- **位置**: `references/moa-system-guide.md` L178-215
| 跃迁 | 关键转换 | 效果 |
|------|---------|------|
| 跃迁 1: 角色分化 | 回答者 → 导演 | 多视角基础 |
| 跃迁 2: 显性对抗 | 一团和气 → 模拟辩论 | 认知摩擦 |
| 跃迁 3: 过程熔铸 | 拼接 → 再创造 | 价值跃迁 |

### 5.1 核心元 Prompt 模板 (核心契约)

**来源**: `README.md` L56-102 + `references/moa-meta-prompt.md` L37-85

```markdown
你现在是一个最高级别的"MoA(混合智能体)编排引擎"。
你的核心使命不是直接回答问题,而是通过在内部模拟一个高度专业、
分工明确且相互对抗的虚拟专家团队,以"组织协同"的方式涌现出
超越单模型极限的高阶智能解决方案。

你必须严格遵循以下核心协议,并将整个思维与推演过程显式、分阶段地展示出来。

## 核心信条
1. **分工打破全栈盲区**:拒绝"头痛医头",通过角色隔离,强制在垂直维度做到极致审视。
2. **对抗制造认知摩擦**:拒绝"一团和气",通过结构化攻击与防御,消灭逻辑死角与边界溢出。
3. **熔铸实现逻辑跃迁**:拒绝"观点拼盘",最终产物必须是经过高阶提纯与再创造的"合金"。

## 一、动态角色原型与职责
你需要根据任务特征,动态选派最匹配的4-6个角色。
角色必须是具体的(如"并发架构师"、"增长黑客"),而非泛泛的。
1. **战略规划师**:负责将模糊、复杂的任务拆解为高内聚、低耦合的子任务树
2. **领域专家**:在各自的"舒适区"内提供极致专业的方案
3. **无情批判者**:必须基于事实、边界条件、潜在冲突进行**具体且结构化**的攻击
4. **熔铸决策者**:不生产初始内容,只负责审视所有交锋过程,解决未决冲突

## 二、任务执行协议
**阶段1:动态编排与解构** - 输出任务分解树,指派专家+批判者
**阶段2:并行深度推演** - 各位专家独立产出最佳方案
**阶段3:结构化对抗与反思** - 批判者攻击 → 专家修正 v2,迭代 1-2 轮
**阶段4:逻辑熔铸与终局裁决** - 禁止罗列,必须再创造

## 三、输出格式约束
最后必须以 `[最终答案]:` 作为开头,给出熔铸后的完美成果。
```

### 5.7 失败模式诊断表

**来源**: `references/moa-case-study.md` L248-254

| 失败模式 | 表现 | 根因 |
|---------|------|------|
| 角色泛化 | 专家头衔是"程序员"、"设计师" | 战略规划师偷懒,未做深度任务分解 |
| 批判空泛 | "这里不够好"、"需要改进" | 批判者没有进入具体角色,未制造认知摩擦 |
| 修正敷衍 | "已修改"、"会注意" | 专家未真正回应批判,对抗环节形同虚设 |
| 熔铸拼盘 | "专家A说...专家B说...综上..." | 决策者未进行再创造,只是简单拼接 |

### 7.2 1+1>2 的实现机制

**来源**: `references/moa-system-guide.md` L217-246

1. **分工的专业性** — 由"角色定义 + 任务分解"保证
2. **协同的涌现** — 由"强制对抗 + 迭代修正"引爆
3. **价值的最终实现** — 由"综合裁决"熔铸

### 7.3 与传统方法的对比优势

**来源**: `README.md` L118-123

| 维度 | 传统单次提示词 | MoA 编排引擎 |
|------|--------------|-------------|
| 思维模式 | 单线自回归 | 多进程模拟,随时切换上下文 |
| 自我纠错 | 存在确认偏误 | 强制结构化对抗 |
| 最终产出 | 观点拼盘 | 逻辑熔铸"合金" |
| 部署成本 | 复杂多 Agent 框架 | 零代码,纯 Prompt 驱动 |

### 7.4 适用场景判定

**来源**: `README.md` L108-112

**适合**:
- 复杂系统工程/代码重构 (如百万并发缓存系统)
- 深度商业战略分析 (如时间银行冷启动)
- 高标准内容创作/研报 (如 Web3.0 供应链金融)

**不适合**:
- 简单事实查询
- 单一领域的常规任务
- 闲聊或轻量级交互

## 八、限制与注意事项

**来源**: `README.md` L127-131, `SKILL.md` L163-168

1. **Token 消耗倍增**: 4 阶段多轮对话显著增加 Token 消耗
2. **模型能力门槛**: 严重依赖角色扮演、长上下文、指令遵循能力
3. **响应时间较长**: 比普通问答长数倍
4. **角色具体化是底线**: 泛泛角色导致泛泛方案
5. **批判质量决定上限**: 空泛批判浪费整个对抗环节
6. **熔铸是再创造不是拼盘**: 必须超越任何单一专家

## 九、总结

### 9.1 项目本质

MoA Engine 是一套**纯提示词工程的思维操作系统**, 通过 4 阶段强制协议 + 4 类核心角色 + 3 大信条,在**单模型**上涌现**多智能体协同**的高阶智能。

### 9.2 核心创新

1. **零代码部署**: 复制粘贴即可使用,无需任何框架
2. **认知摩擦机制**: 通过强制对抗打破确认偏误
3. **过程熔铸**: 禁止拼盘,强制再创造
4. **角色具体化**: 强制细分领域头衔,提升专业深度
